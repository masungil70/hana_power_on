# 1. Event 구조란 무엇인가?

**Event 구조(Event Log 구조)**는 금융 데이터를 이해하는 데 가장 중요한 개념 중 하나입니다. 단순히 “테이블 데이터”로 보는 관점을 넘어, **모든 금융 데이터를 ‘사건(이벤트)’의 연속으로 해석하는 방식**입니다.

금융 데이터에서 “한 줄(row)”은 단순 데이터가 아니라 다음과 같습니다:

> “누가, 언제, 어디서, 무엇을, 얼마에, 어떤 방식으로 행동했는가”라는 하나의 사건

즉, 금융 데이터는 **기록 데이터가 아니라 행동 로그(Event Log)** 입니다.

---

## 📌 예시

```
A 고객이 B 가맹점에서 120달러 결제
```

이 한 문장을 구조화하면:

| 요소            | 의미             |
| --------------- | -----------      |
| Actor (행위자)  | A 고객           |
| Action (행동)   | 결제             |
| Target (대상)   | B 가맹점         |
| Amount (금액)   | 120달러          |
| Time (시간)     | 특정 시점        |
| Context (상황)  | 카드 결제 시스템 |

---

# 2. 왜 “Event 구조”가 중요한가?

금융 데이터는 단일 값이 아니라 **행동의 흐름**이기 때문입니다.

---

## ❌ 잘못된 관점 (단순 테이블)

* A 고객 = 120달러
* B 가맹점 = 음식점
* Fraud = 0

👉 이건 의미를 잃은 데이터

---

## ✅ 올바른 관점 (Event 구조)

* A 고객이
* 특정 시간에
* 특정 장소에서
* 특정 금액으로
* 특정 행동을 했다

👉 “행동 시나리오”가 된다

---

# 3. Event 구조의 핵심 구성 요소

금융 Event는 보통 5가지 요소로 구성됩니다.

---

## 3-1. Actor (행위자)

> “누가 했는가?”

예:

* 고객 (Customer)
* 계좌 (Account)
* 카드 (Card Number)

📌 핵심:

* 모든 이벤트는 반드시 “주체”가 있음

---

## 3-2. Action (행동)

> “무슨 일이 발생했는가?”

예:

* 결제 (Payment)
* 입금 (Deposit)
* 출금 (Withdrawal)
* 로그인 (Login)

📌 핵심:

* 금융 데이터의 “동사”

---

## 3-3. Object (대상)

> “무엇에 대해 행동했는가?”

예:

* 가맹점 (Merchant)
* 계좌 (Account)
* 상품 (Product)

📌 핵심:

* 행동의 방향성 존재

---

## 3-4. Context (맥락)

> “어떤 상황에서 발생했는가?”

예:

* 위치 (GPS)
* 시간 (새벽/주간)
* 디바이스 (모바일/PC)

📌 핵심:

* Fraud 판단 핵심 요소

---

## 3-5. Attributes (속성)

> “얼마나, 어떤 조건으로 발생했는가?”

예:

* 금액 (amt)
* 통화 (USD/KRW)
* 수수료

---

# 4. Event 구조를 실제 데이터로 변환하면

## 원본 문장

```
A 고객이 B 가맹점에서 120달러 결제
```

---

## Event Log 구조 (JSON 형태)

```json id="ev1"
{
  "actor": "A 고객",
  "action": "payment",
  "object": "B 가맹점",
  "amount": 120,
  "currency": "USD",
  "timestamp": "2026-05-11 21:30:00",
  "location": "Seoul"
}
```

---

👉 이 구조가 바로 **금융 데이터의 본질**

---

# 5. Event 구조가 중요한 이유 (핵심 4가지)

---

## 5-1. “시간 흐름” 분석 가능

Event는 항상 순서를 가짐:

```
Login → Search → Add to Cart → Payment
```

👉 금융에서는:

```
결제 → 결제 → 결제 → 해외결제
```

📌 의미:

* 행동 패턴 분석 가능

---

## 5-2. 이상 탐지 (Fraud Detection)

Event 구조 없으면 Fraud를 못 잡습니다.

### 예:

```
09:00 한국 → 09:05 미국 결제
```

👉 물리적으로 불가능 → 이상 거래

---

## 5-3. 행동 패턴 모델링

고객 행동은 “이벤트 시퀀스”입니다:

```
식당 → 편의점 → 온라인쇼핑 → 해외결제
```

👉 고객 라이프스타일 분석 가능

---

## 5-4. 시계열 + 관계 분석 가능

Event는 단순 데이터가 아니라:

* 시간 (Time Series)
* 관계 (Graph)
* 행동 (Sequence)

3가지가 결합된 구조입니다.

---

# 6. 금융 데이터 = Event Graph

Event 구조를 확장하면 이렇게 됩니다:

```
Customer A
   ├── payment → Merchant X
   ├── payment → Merchant Y
   ├── login → Mobile App
```

👉 즉 금융 데이터는 “그래프 데이터”

---

# 7. Event 구조 vs 일반 테이블 구조 비교

| 구분 | 일반 테이블  | Event 구조         |
| ---- | ------------ | ------------------ |
| 형태 | 정적 데이터  | 동적 행동          |
| 중심 | 값           | 사건               |
| 분석 | 통계         | 패턴               |
| 목적 | 저장         | 이해               |
| 예   | amt=120      | A가 B에서 120 결제 |

---

# 8. 핵심 정리

금융 데이터의 Event 구조는 이렇게 이해하면 됩니다:

> “금융 데이터는 숫자의 집합이 아니라, 사람과 시스템이 시간 속에서 만들어낸 행동 기록(Event Log)이다.”

그리고 하나의 거래는:

* 누가 (Actor)
* 무엇을 (Action)
* 어디에 (Object)
* 어떤 상황에서 (Context)
* 얼마를 (Attribute)

수행한 **하나의 사건**입니다.
